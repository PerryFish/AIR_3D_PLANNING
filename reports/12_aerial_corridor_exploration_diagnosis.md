# Aerial Corridor Exploration Diagnosis

## Problem

The previous visual demo behaved like a high-altitude overflight. The UAV body, green breadcrumb trail, and yellow goal marker could appear above most or all dense50 obstacles, which did not match the required 3D aerial corridor exploration behavior.

Correct behavior is aerial navigation among obstacles: the UAV remains airborne, but its state, goals, and trail stay in the available 3D free space between obstacles rather than above the full obstacle field.

## Diagnosis

- dense50 obstacle height: `0.7 m` to `2.5 m`, generated by `scripts/generate_dense50_gazebo_world.py`.
- original UAV initial z: `1.5 m`.
- original `/way_point` or `/aerial_exploration/goal` z:
  - raw frontier voxel z could be `0.5, 1.5, 2.5, 3.5, 4.5, 5.5 m`;
  - fallback lawnmower goal was hardcoded to `2.5 m`.
- original Gazebo UAV visual z: `/state_estimation.z + 0.25 m`.
- original green breadcrumb z: `/state_estimation.z + 0.05 m`.
- original yellow goal marker z: goal z without corridor clamping.
- original planner behavior: frontier selection used 3D voxel coordinates directly and the path was a straight interpolation to that z.
- original limitation: this was not obstacle-aware 3D corridor planning; it could become XY/free-frontier selection plus arbitrary voxel z.
- coverage calculation: coverage is still based on simulated pose and sensor-range observation of the voxel map. It is suitable for this MVP simulation loop, but it is not a full real sensor pipeline.

## Fix

The exploration loop is now constrained to an aerial corridor:

- `aerial_corridor_min_z: 0.8`
- `aerial_corridor_default_z: 1.4`
- `aerial_corridor_max_z: 2.2`
- `aerial_corridor_clearance: 0.25`
- `avoid_flying_above_obstacle_top: true`
- `max_above_obstacle_margin: 0.2`

Planner changes:

- Frontier XY is reused, but goal z is set to `aerial_corridor_default_z`.
- Goal z and path z are clamped to `[0.8, 2.2]`.
- Candidate goals inside occupied 3D voxels at the corridor layer are rejected.
- Segment collision is sampled in 3D against the dense50 voxel occupancy.
- If a straight segment is blocked, a small XY A* route is generated through free corridor cells at the aerial corridor z.
- The dense50 grid model now matches the Gazebo world generator, including protected free corridors and the `0.7-2.5 m` obstacle height profile.

Follower and Gazebo visualization changes:

- `simple_uav_follower_node` starts at `1.4 m` and clamps path targets to `[0.8, 2.2]`.
- `gazebo_uav_visualizer` uses true `/state_estimation.z` by default: `visual_z_offset=0.0`.
- `gazebo_trail_visualizer` uses true pose and goal z by default: `trail_z_offset=0.0`, `goal_z_offset=0.0`.
- The simple UAV model was reduced in size so it is visually appropriate for obstacle corridors.

## Current Planner Scope

This is still an MVP corridor planner, not a full kinodynamic UAV planner. It performs:

- 3D occupied-voxel collision checking at the selected aerial corridor layer;
- XY A* through free-space corridor cells;
- fixed aerial z at `1.4 m` with clamped z bounds.

It does not yet perform full 3D multi-altitude A* or Gazebo physics flight control.

## Verification Target

The validation scripts check that:

- `/state_estimation.z` remains in `[0.8, 2.2]`;
- `/aerial_exploration/goal.z` remains in `[0.8, 2.2]`;
- robot and goal z are not above `2.5 + 0.2 m`;
- Gazebo visual offsets are zero by default;
- coverage still reaches `>= 0.93` with `done=True`.
- In the sensor mapping baseline, the corridor planner consumes frontier cells from the online observed map rather than only the legacy synthetic radius map.
